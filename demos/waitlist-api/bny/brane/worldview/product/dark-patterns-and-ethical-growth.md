# Dark Patterns and Ethical Growth
Waitlist referral mechanics sit on a spectrum from genuine value exchange to manipulative coercion — the API's defaults shape which side founders land on.

## Common Dark Patterns in Waitlists

### Artificial Scarcity
"Only 500 spots left!" when there's no actual capacity limit. The waitlist exists purely for hype. The API enables this by design — should it care?

### Hidden Regression
User is #50, refers nobody, silently drops to #200 as others refer. The user sees no movement and assumes the system is broken. Some implementations hide position entirely to avoid this — but that removes the core incentive to refer.

### Guilt Sharing
"Share with friends or lose your spot" — coerced sharing generates resentment. The referrals feel spammy to recipients, harming the founder's brand.

### Phantom Progress
Showing fake position improvements ("You moved up 12 spots!") to maintain engagement. Technically the API doesn't do this, but founders could abuse position framing endpoints.

### Endless Waitlist
Some products never actually launch or promote anyone. The waitlist becomes a lead-gen tool disguised as access management.

## Ethical Defaults the API Can Set
- Show honest position context (even if framed positively)
- Require that referral rewards are disclosed to users
- Provide founders with templates for honest status emails
- Never auto-send from users' accounts or address books
- Make unsubscribe/deletion trivial

## The Platform Responsibility Question
Is the API responsible for how founders use it? Technically no — but if the API makes dark patterns easy and ethical patterns hard, the default outcome is manipulation. The API's design choices ARE ethical choices.

## A Differentiator
Positioning as "the ethical waitlist API" could be a market angle. Founders who care about their brand want tools that don't make them look sleazy. Trust badges, transparency features, and honest defaults could justify premium pricing.
