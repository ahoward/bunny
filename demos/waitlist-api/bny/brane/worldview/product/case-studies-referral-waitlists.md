# Case Studies: Referral Waitlists
Real-world waitlist launches reveal patterns that theory alone can't predict — what actually happened when companies bet on referral-driven queues.

## Robinhood (2014)
- 1 million signups before launch
- Mechanic: refer friends → move up the queue. No specific threshold, continuous improvement.
- Key insight: showed exact position AND total count. "You are #487,223 of 981,001" created FOMO and urgency simultaneously.
- What worked: the product itself was revolutionary (free stock trading), so the waitlist amplified genuine demand.
- What to learn: referral mechanics amplify existing desire, they don't create it.

## Superhuman (2019)
- Waitlist ran for years, not weeks
- Mechanic: referral from existing user required (not just queue position)
- Created genuine exclusivity, not manufactured scarcity
- Key insight: the long wait became a feature, not a bug — it was a status signal
- What to learn: for premium positioning, slower promotion can increase perceived value

## Clubhouse (2020)
- Invite-only with limited invites per user
- Invites became social currency — people traded them
- The scarcity was real (server capacity) but also strategic
- Downside: when they opened up, the exclusivity premium collapsed and so did engagement
- What to learn: manufactured scarcity has an expiration date

## Hey.com (2020)
- Simple email waitlist with position shown
- No referral mechanic — relied on brand (Basecamp) and controversy (App Store fight)
- What to learn: sometimes the story IS the viral mechanic, not the referral system

## Lessons for the API
1. The product's inherent desirability determines referral effectiveness — the API can't fix a boring product
2. Position visibility is powerful but must be carefully framed
3. Referral mechanics should match the product's brand (playful vs. exclusive vs. urgent)
4. The API should support multiple referral strategies, not bake in one model
